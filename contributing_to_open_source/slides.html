Open Source vs. Community

Quick distinction: OS is about the code, community is about the people (and the code)!

Why?

  • Building confidence
  • Developer experience
  • Professional opportunities
  • Community provides all this and more

Types of Contributions

  • Features

Just because you groom an issue, doesn't mean you can't contribute a fix later. I would argue this is all part of "the contribution life cycle"

Tips for Finding a Project

  1. Find a project you already use
  2. Don't tackle BIG projects on your first try
  3. Find supporting projects like plugins to start off
It makes it a lot easier if you're familiar with the workings of a project as a user, it'll give you an intuition High profile projects with lots of developers and activity will give you a hard time establishing a reputation With time and experience you'll become comfortable with tackling larger projects

Reading Code

Reading someone else's code can seem daunting

Most of a developer's life will be spent reading code

“Indeed, the ratio of time spent reading versus writing is well over 10 to 1. We are constantly reading old code as part of the effort to write new code. ...[Therefore,] making it easy to read makes it easier to write.”
-- Robert C. Martin

Code Reading Tips

  • Take "stream of conciousness" notes
  • Learn to use grep/IDE's advanced search features
  • Read test cases, they offer a lot of context

Preparing Your Change

A lot of projects will provide guidance on how they like changes to be submitted

  • CONTRIBUTING(.txt)
  • README(.txt)
  • Project wiki/docs
Finding/reading/implementing these suggestions before submitting is REALLY important, don't waste the maintainers time

Specific Prep Steps

  • Tests
  • Documentation
  • Linting
  • Context in PR description
  • Backporting
  • Contributor license agreement

Getting Your Change Accepted

Make it as little work as possible for the maintainer to accept it

Polite persistence is the key

Getting constructive feedback

Expect scrutiny

Make it clear what your change is about, Make sure your code is clean and easy to validate, After that they shouldn't have a problem merging it Remember that you're trying to build a rapport here

Dealing With Rejection

  • Is your feature considered out of scope?
  • Did you misunderstand the project's target use case?
  • Toxic community?
  • Get back on the horse!
